# Week 49: Observability and the Three Pillars

Today I began Month 13, transitioning from infrastructure provisioning to day-two operations: Monitoring and Observability.

## Monitoring vs. Observability
* **Monitoring** is the act of collecting data to track the overall health of a system and alert on predefined thresholds. It tells you *when* something goes wrong (e.g., CPU > 90%).
* **Observability** is a measure of how well internal states of a system can be inferred from its external outputs. It provides the context needed to understand *why* something went wrong in highly complex, distributed microservices.

## The Three Pillars
True observability relies on triangulating data across three dimensions:
1.  **Metrics:** Quantitative, time-series data representing the overall health and performance of the system over time (e.g., memory usage, error rates).
2.  **Logs:** Discrete, timestamped text records of specific events or errors generated by the application code.
3.  **Traces:** Distributed tracking of a single user request as it traverses across multiple microservices and network hops.

## Focus on Metrics
Metrics form the foundation of proactive alerting. Because they are strictly numeric (Counters, Gauges, Histograms) bound to a timestamp, they require minimal storage overhead compared to logs. This time-series format allows DevOps engineers to mathematically query the data to detect anomalies, such as sudden spikes in 500-level HTTP responses, before a total system outage occurs.
